## 下次测试：文件与操作

1. 打开“许可证签发器”，在“设备码”中粘贴用户或本机支持者窗口显示的完整 `QF1-…`。
2. 对本机联调，点击“从文件导入”并选择 `tools/license-cli/local-test/private.key`；或者仅复制该文件的 44 字符 Base64 内容到“签名私钥”输入框。**不要选择** `public-key.txt` 或已有的 `*.license.json`。
3. 选择对应皮肤，点击“生成许可证”；复制生成的完整 JSON。
4. 回到“支持者皮肤”窗口，把 JSON 粘贴到“粘贴许可证”，点击“验证许可证”。验证成功后从托盘“主题 → 支持者皮肤”选择已解锁皮肤。
5. 重启应用，确认皮肤仍保留；再用另一台机器的设备码签发或尝试粘贴，确认跨设备许可证被拒绝。

## 正式分发密钥规则（发布阻塞）

- 使用一对新的、离线保存的 **Ed25519** 密钥：私钥为 Base64 编码的 32 字节 seed，公钥为 Base64 编码的 32 字节验证密钥。
- 维护者签发时使用正式私钥；它只保存在离线加密介质或受控签发机器上，绝不放入仓库、安装包、前端变量、聊天记录或发给用户。
- 用户分发包只编译注入正式公钥：构建时设置 `QUOTA_FLOAT_LICENSE_PUBLIC_KEY` 为正式 `public-key.txt` 的内容。公钥可以随安装包分发；没有该公钥，用户端无法验证许可证。
- `tools/license-cli/local-test/private.key` 和 `public-key.txt` 仅用于当前联调；正式发布前必须新生成一对正式密钥，且必须用正式公钥重新打包。正式包不能接受本地测试私钥签发的许可证。

## 当前结论

项目已完成“离线许可证解锁支持者皮肤”的主要代码；交易采用小红书或闲鱼线下完成，不在应用内接入二维码、支付链接、订单系统或自动发货。此前考虑过二维码流程，因安全性与实现成本的性价比不合适而未采用。

## 页面与调用链

```text
托盘“主题” → “支持者皮肤” → supporter 独立窗口
  → getSupporterStatus：读取偏好与本机 QF1 请求码
  → 用户复制请求码，在小红书/闲鱼与维护者线下交易
  → 维护者 CLI 使用离线私钥签发许可证 JSON
  → 用户粘贴 JSON，点击“验证许可证”
  → activate_supporter_license：检查版本、皮肤、设备码、keyId、Ed25519 签名
  → 保存 license / unlockedSkin / selectedSkin，通知 widget
  → App 仅在两个 skin 字段均为 blur 时渲染 Blur 外观
```

- `src/main.tsx`：有 `?supporter` 时渲染支持者页。
- `src/components/SupporterPanel.tsx`：显示请求码、复制、粘贴/验证许可证、选择已解锁皮肤与设计师 ID `@Change`。
- `src-tauri/src/license.rs`：将 Windows `MachineGuid` 或 macOS `IOPlatformUUID` 与固定 salt 做 SHA-256，生成不暴露原始机器 ID 的 `QF1-...` 请求码。
- `src-tauri/src/lib.rs`：每次读取状态与选择 Blur 时都重新验签；授权失效时自动回退 `default`。
- 托盘的“主题”下，“默认皮肤”子菜单保留跟随系统/深色/浅色；“支持者皮肤”单独打开支持者页。因此恢复默认外观应在托盘选择默认皮肤，而非在支持者弹窗另设按钮。

## 许可证约定

- 仅接受 `version: 1`、`skinId: "blur"`、`keyId: "supporter-v1"`。
- 签名载荷为 `version`、`skinId`、`deviceHash`、`issuedAt`、`licenseId`、`keyId` 按换行拼接；`signature` 不参与签名。
- 没有过期、撤销、订单查询或联网验证；私钥只供离线 CLI 使用，绝不能放入仓库、Vite 变量或应用包。

## 面向用户的故障处理与安全边界

用户可自行处理的情况只有三类：重新粘贴完整许可证 JSON、重新复制并发送当前设备码、更新到兼容的新版本应用。设备码不匹配、签名无效、未配置公钥或不受支持的皮肤/签名密钥，均由维护者在离线签发与发布环境中处理。

维护者可用 `tools/license-cli` 重新签发许可证并在验证机器上做端到端核对，但验证机器不能替代正式发布构建中的公钥配置。为避免盗用：私钥与完整许可证绝不交给用户；许可证必须绑定用户提供的请求码；每次换机均重新签发；台账仅记录请求码前缀和许可证 ID，不记录原始硬件 ID。

## Blur 皮肤素材边界

Blur 仅在桌面端验签后由 `App.tsx` 传入 `skin: "blur"`，默认皮肤的配色、结构与状态不受影响。三个状态背景与按钮底图由 Blur 专属 CSS 使用；不可用、过期、未登录状态仍回退默认浅色皮肤。所有离线桌面包内的图片都可能被具备技术能力的人提取，因此许可证能保护“正常使用与解锁入口”，不能把本地素材变成不可复制的 DRM。若未来需要真正限制原始素材分发，必须改为联网、在授权后下载资源，并接受服务端与账号体系的成本。
